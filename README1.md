# Задача 1: Видача решти — жадібний алгоритм vs динамічне програмування

## Опис задачі

Реалізовано дві функції для касової системи, що видає решту з набору монет `[50, 25, 10, 5, 2, 1]`:

| Функція | Підхід |
|---|---|
| `find_coins_greedy(amount)` | Жадібний алгоритм |
| `find_min_coins(amount)` | Динамічне програмування |

---

## Приклад роботи (сума = 113)

```
Жадібний алгоритм : {50: 2, 10: 1, 2: 1, 1: 1}
Динамічне програм.: {1: 1, 2: 1, 10: 1, 50: 2}
```

Обидва алгоритми повертають однаковий результат (5 монет) — для цього набору номіналів жадібний алгоритм також знаходить оптимальне рішення.

---

## Аналіз складності (O-велике)

### Жадібний алгоритм
- **Часова складність: O(n)** — один прохід по фіксованому списку номіналів (`n = 6`). Фактично це **O(1)** — константний час, незалежний від суми.
- **Просторова складність: O(n)** — словник результату містить щонайбільше `n` записів.

### Динамічне програмування
- **Часова складність: O(S × n)** — де `S` — сума, `n` — кількість номіналів. Для `S = 1 000 000` і `n = 6` — це **6 мільйонів операцій**.
- **Просторова складність: O(S)** — два масиви розміром `S+1`.

## Висновки

### Коли жадібний алгоритм є кращим?

Для **стандартних касових систем** (де набір монет чітко спроєктований так, що жадібний підхід завжди дає оптимум) жадібний алгоритм є беззаперечно кращим:

- Виконується за **константний час O(1)** незалежно від суми.
- Простий у реалізації та підтримці.
- Підходить для обробки мільйонів транзакцій у реальному часі.

### Коли потрібне динамічне програмування?

ДП гарантує **мінімальну кількість монет** для **будь-якого** набору номіналів, включно з екзотичними наборами, де жадібний алгоритм дає субоптимальний результат.

**Приклад:** набір монет `[1, 3, 4]`, сума `6`:
- Жадібний: `4 + 1 + 1 = 3 монети`
- ДП (оптимум): `3 + 3 = 2 монети`

### Підсумок

| Критерій | Жадібний | ДП |
|---|---|---|
| Швидкість |  O(1) |  O(S×n) |
| Гарантія оптимуму |  Тільки для певних наборів монет | Завжди |
| Використання пам'яті |  O(n) |  O(S) |
| Великі суми | Без проблем |  Повільно |
| Довільні номінали | Може бути неоптимальним | Завжди коректний |

**Для набору `[50, 25, 10, 5, 2, 1]` жадібний алгоритм завжди знаходить оптимальне рішення** — цей набір є «канонічним» (кожен більший номінал кратний меншим або утворює покриття без втрат). Тому в реальній касовій системі з таким набором монет жадібний алгоритм є оптимальним вибором як за швидкістю, так і за якістю результату.
